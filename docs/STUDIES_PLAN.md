# ZERTZ Studies — план фичи (аналог Lichess Studies)

> Живой документ. Обновлять статус фаз по мере продвижения.
> Ветка: **`feature/studies`** (ответвлена от `main`). UI не мержить в main, пока фича не собрана.

## Цель

Интерактивные учебники по ZERTZ. Автор создаёт «студию» (доска + дерево ходов +
markdown-комментарии), другие открывают её как интерактивный учебник: свободно
листают ветки, тыкают позиции — но **правки читателя не сохраняются**. Править
может только владелец. Любую студию можно склонировать («форкнуть») к себе.
Отдельный экран в разделе меню **«Обучение»**.

## Зафиксированные решения (НЕ переобсуждать без явной просьбы)

1. **Иерархия — только узлы, без папок.** Каждый узел = одновременно «страница»
   (свой контент) И может иметь детей (Notion-подобно). Бесконечная вложенность.
2. **Оба входа создания студии:** (а) со стандартной стартовой позиции + прокликать
   ходы; (б) произвольный **редактор позиции** — нужен на старте (много партий с
   другой платформы, откуда нельзя импортировать; вводят вручную).
3. **Комментарии — markdown** (переиспользуем блоговый рендер: `react-markdown` + `prose`).
4. **Публичный список** чужих студий — в разделе «Обучение».
5. **Баланс шариков в редакторе — жёсткий инвариант.** Валидация только
   баланс + структура (шарики лишь на существующих кольцах); зертц-легальность
   (свободные кольца/изоляции) НЕ проверяем — иначе не ввести оборванные позиции.
   Инвариант: `reserve + на_доске + captures(p1) + captures(p2) == {6 белых, 8 серых, 10 чёрных}`.
6. **Один владелец.** Соавторства нет — при надобности клонируют себе и форкают.
7. **URL:** `/studies/:owner/:slug`. Плоский (иерархия — только в сайдбаре, не в пути).
   Слаг: транслит из тайтла по умолчанию, но **правится вручную и всегда изменяем**.
   Протухшие при переименовании ссылки — забота автора (осознанно приняли).

## Модель данных

Таблица `studies` (создаётся в `server/db.js` → `ensureSchema`, idempotent):

| колонка | тип | назначение |
|---|---|---|
| `id` | SERIAL PK | |
| `owner_id` | INT → users(id) CASCADE | владелец |
| `parent_id` | INT → studies(id) CASCADE | NULL = корень; иерархия |
| `slug` | TEXT | уникален у владельца, изменяем; в URL |
| `title` | TEXT | |
| `board_size` | SMALLINT | 37/48/61 |
| `setup_json` | TEXT | стартовый `GameState` (стандартный или из редактора) |
| `tree_json` | TEXT | дерево `GameNode` (+ поле `comment` на узлах) |
| `root_comment` | TEXT | markdown-вступление к узлу |
| `meta_json` | JSONB | `{players:{white,black}, event, round, timeControl, date, result}` (опц.) |
| `is_public` | BOOL | |
| `sort` | DOUBLE PRECISION | fractional index → drag-reorder меняет один узел |
| `created_at` / `updated_at` | TIMESTAMP | UTC (см. конвенцию таймстемпов в CLAUDE.md) |

Индексы: `UNIQUE(owner_id, slug)`, `(owner_id, parent_id, sort)`, partial `WHERE is_public`.

`GameNode` расширен полем `comment?: string` (`src/game/types.ts`). `serializeTree`/
`deserializeTree` тащат его автоматически (spread `{...n}`) — отдельной сериализации не нужно.

## API (`/api/studies`, `server/routes/studies.js`)

| Метод | Путь | Права | Назначение |
|---|---|---|---|
| GET | `/tree` | authRequired | Своя иерархия для сайдбара (без контента: id/parentId/slug/title/sort/isPublic) |
| GET | `/public?limit&offset` | — | Публичный список (ник владельца + title + slug) |
| GET | `/:owner/:slug` | optionalAuth | Полный контент узла + сводка прямых детей; 403 если приватно и не владелец |
| POST | `/` | authRequired | Создать `{parentId?, title, boardSize, setupJson, treeJson, rootComment?, meta?, isPublic?, slug?}` |
| PUT | `/:id` | владелец | Обновить контент (COALESCE-патч; comment/meta через boolean-флаг «прислано ли поле») |
| PUT | `/:id/move` | владелец | Reparent/reorder `{newParentId, sort?}`; защита от циклов (рекурсивный CTE) |
| DELETE | `/:id` | владелец | Удалить поддерево (FK cascade) |
| POST | `/:id/clone` | authRequired | Глубокая копия читаемого поддерева себе (private, слаги регенерятся, транзакция) |

Клиент: `src/db/studiesApi.ts` — типы (`StudyNode`, `StudyTreeNode`, `PublicStudy`, …) + обёртки.

## Ключевые архитектурные заметки

- **Права читателя.** Читатель никогда не шлёт PUT. Его исследование веток живёт
  только в клиентском стейте (рабочий оверлей поверх авторского дерева) + кнопка
  «вернуться к линии автора». Реализуется на Этапе C/E.
- **Ленивая загрузка иерархии.** Сайдбар грузит только `/tree` (без контента).
  Контент узла — по клику через `/:owner/:slug`.
- **Синхронизация не нужна.** Студии — статичные уроки; live-sync как в Lichess не
  делаем (у нас и так поллинг). Проще.
- **Переиспользование:** дерево `GameNode`, режим анализа (игра вариантов → ветки),
  вертикальное дерево из премувов (`RoomScreen` Plan-tab), панель нотации,
  `serializeTree`/`deserializeTree`, `react-markdown` из блога, спектейт как база read-only.

## Gotchas / на что не наступить

- **Баланс шариков.** Всего в игре `6 белых, 8 серых, 10 чёрных`. Редактор (Этап D)
  обязан держать инвариант, иначе win-check и уроки едут.
- **`state.moveNumber` стартует с 1** (см. CLAUDE.md) — обычная off-by-one ловушка.
- **Кодировка в curl-тестах на Windows.** Bash/curl шлёт кириллицу в CP1251 → сервер
  видит битый UTF-8 → слаг падает в фолбэк `study`. Это артефакт теста, НЕ баг:
  `slugify` на настоящем UTF-8 работает (`Дебютные ловушки → debyutnye-lovushki`).
  Браузерный `fetch`+JSON шлёт корректный UTF-8.
- **Бэкенд без hot-reload** (`node server/server.js`) — после правок в `server/**`
  перезапускать вручную.
- **Dev-БД:** остались тестовые строки от смоук-теста (юзер `tester_*`, пара студий
  с битыми title). Безвредны, можно подчистить.

## На будущее (не сейчас)

- **Авто-проверка задач.** Пользователь хочет пазлы с авто-проверкой. Для этого
  НЕ нужна нейросеть — нужен **точный solver/эндшпильная база** (есть ли форсированное
  и единственное решение). Это перебор, а не обучение.
- **Self-play RL (AlphaZero-lite).** Возможен (MCTS + policy/value сеть, симулятор
  уже есть в `GameEngine`/`replay.js`), но упирается в GPU-компьют и энкодинг
  составного хода (постановка + снятие кольца + цепочки взятий). Отдельная большая тема.
